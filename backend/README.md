# Backend Documentation

## Overview
This document provides an overview of the backend architecture, its components, and how they interact to support the portfolio application.

## Architecture
The backend is built using the following technologies:
- **Node.js**: JavaScript runtime for server-side execution.
- **Express**: Web framework for handling HTTP requests and routing.
- **Prisma**: ORM (Object-Relational Mapping) for database interactions.
- **PostgreSQL**: Relational database for storing project data.

## Key Files and Their Roles

### 1. `package.json`
- **Purpose**: Defines the project's dependencies, scripts, and metadata.
- **Key Dependencies**:
  - `express`: Web framework for handling HTTP requests.
  - `@prisma/client`: Prisma client for database interactions.
  - `cors`: Middleware for enabling CORS (Cross-Origin Resource Sharing).
  - `dotenv`: Loads environment variables from a `.env` file.
- **Scripts**:
  - `dev`: Starts the server in development mode using `ts-node-dev`.
  - `build`: Compiles TypeScript code to JavaScript.
  - `start`: Starts the compiled server.
  - `prisma:generate`: Generates the Prisma client.
  - `prisma:migrate`: Applies database migrations.
  - `prisma:studio`: Opens Prisma Studio for database management.
  - `prisma:seed`: Seeds the database with initial data.

### 2. `tsconfig.json`
- **Purpose**: Configures TypeScript compilation options.
- **Key Settings**:
  - `target`: Compiles to ES2020.
  - `module`: Uses CommonJS module system.
  - `outDir`: Output directory for compiled JavaScript files.
  - `rootDir`: Source directory for TypeScript files.
  - `strict`: Enables strict type-checking options.

### 3. `src/app.ts`
- **Purpose**: Configures the Express application.
- **Key Features**:
  - Initializes Express and Prisma.
  - Configures middleware for CORS and JSON parsing.
  - Sets up routes for the API.

### 4. `src/server.ts`
- **Purpose**: Starts the Express server.
- **Key Features**:
  - Listens on a specified port (default: 5000).
  - Handles port conflicts by attempting to use the next available port.

### 5. `prisma/schema.prisma`
- **Purpose**: Defines the database schema using Prisma's declarative syntax.
- **Key Features**:
  - Configures the Prisma client generator.
  - Specifies the database provider (PostgreSQL).
  - Defines the `Project` model with fields such as `title`, `description`, `tags`, `urlRepo`, `urlDemo`, `coverImage`, `images`, `createdAt`, and `updatedAt`.

### 6. `src/controllers/projectsController.ts`
- **Purpose**: Handles business logic for project-related operations.
- **Key Features**:
  - Implements the `getProjects` function to fetch all projects from the database.
  - Uses Prisma to interact with the database.

### 7. `src/models/Project.ts`
- **Purpose**: Placeholder for the Project model.
- **Key Features**:
  - Notes that the Project model types are generated by Prisma.

### 8. `src/routes/projects.ts`
- **Purpose**: Defines routes for project-related endpoints.
- **Key Features**:
  - Sets up a GET route to fetch all projects.
  - Uses the `getProjects` controller function.

## How It Works

1. **Server Initialization**:
   - The server is started using `src/server.ts`, which initializes the Express app defined in `src/app.ts`.
   - The server listens on the specified port and handles port conflicts gracefully.

2. **API Endpoints**:
   - The `/api/projects` endpoint is defined in `src/routes/projects.ts` and handled by the `getProjects` function in `src/controllers/projectsController.ts`.
   - This endpoint fetches all projects from the database using Prisma.

3. **Database Interaction**:
   - Prisma is used to interact with the PostgreSQL database.
   - The `Project` model is defined in `prisma/schema.prisma`, and Prisma generates the necessary TypeScript types for this model.

4. **Middleware**:
   - CORS is enabled to allow cross-origin requests from the frontend.
   - JSON parsing is configured to handle JSON payloads in requests.

## Summary
The backend is a simple yet robust Express server that provides an API for fetching project data. It uses Prisma for database interactions and is configured to work seamlessly with the frontend of the portfolio application. The architecture is modular, with clear separation of concerns between routes, controllers, and models.
